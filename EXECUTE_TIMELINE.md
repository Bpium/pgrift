## Сценарий выполнения миграции

Кратко: `runner.ts` оркестрирует миграцию, `migrate-tenant.ts` переносит одну БД в схему,
`verify-migration.ts` проверяет результат, а `migration-state.json` позволяет безопасно
останавливать и продолжать процесс.

---

## 1. `runner.ts` — точка входа / оркестратор

- **Подтверждение запуска**
  - Спрашивает у пользователя `Run migration? (Y/N)` и выходит, если ответ не `Y`.
- **Подготовка окружения**
  - Проверяет, что целевая БД (`CONFIG.target.database`) существует.
  - Создаёт директорию дампов (`CONFIG.dumpDir`), если её ещё нет.
- **Определение списка тенантов**
  - Загружает все базы данных с исходного сервера (кроме служебных и исключённых).
  - Читает `migration-state.json` и выкидывает уже завершённые БД.
- **Пакетная миграция**
  - Делит оставшиеся БД на батчи по `CONCURRENCY` штук.
  - Для каждого батча:
    - Логирует прогресс (сколько всего, сколько сделано, ETA).
    - Запускает миграцию каждой БД через `migrateTenant` в `Promise.allSettled`.
    - После батча обновляет и сохраняет состояние (`saveState`).

---

## 2. `migrate-tenant.ts` — миграция одной БД

Для одной базы `dbName` выполняется:

1. **Завершение соединений с исходной БД**
   - В `postgres` вызывается `pg_terminate_backend` для всех сессий, подключённых к `dbName`.
2. **Временное переименование схемы в исходной БД**
   - `ALTER SCHEMA public RENAME TO "dbName"`.
   - `CREATE SCHEMA public` (новый пустой `public`).
   - `ALTER DATABASE "dbName" SET search_path = '"dbName"'`.
3. **Дамп переименованной схемы**
   - `pg_dump -n "dbName"` в файл `${dbName}.final.dump` в `CONFIG.dumpDir`.
4. **Откат изменений в исходной БД**
   - `ALTER DATABASE "dbName" RESET search_path`.
   - `DROP SCHEMA public` (пустой).
   - `ALTER SCHEMA "dbName" RENAME TO public`.
5. **Подготовка целевой БД (tenants)**
   - Подключается к `CONFIG.target`.
   - `DROP SCHEMA IF EXISTS "dbName" CASCADE` (идемпотентность повторных запусков).
   - Создаёт необходимые расширения из списка исходной БД (если ещё не установлены).
6. **Патч дамп-файла**
   - Ищет в дампе `dbName.gin_trgm_ops` и заменяет на `public.gin_trgm_ops`.
7. **Восстановление дампа в целевую БД**
   - Запускает `psql -f "${dbName}.final.dump"` против `CONFIG.target.database`.
8. **Уборка**
   - Удаляет временный дамп-файл.
   - В случае ошибки пытается откатить схему в исходной БД обратно в `public`.

---

## 3. `verify-migration.ts` — проверка после каждой БД

Для каждой успешно мигрированной БД:

- Подключается к исходной БД (`dbName`, схема `public`) и к целевой (`tenants`, схема `"dbName"`).
- **Список таблиц**
  - Сравнивает список таблиц в `public` и `"dbName"`:
    - если какие-то таблицы отсутствуют в целевой схеме — фиксирует ошибку.
- **Кол-во строк**
  - Для каждой таблицы сравнивает `COUNT(*)` в исходной и целевой схеме.
- **Чек-суммы (опционально)**
  - Для таблиц с количеством строк ≤ `SKIP_CHECKSUM_ABOVE_ROWS` (если задано) считает
    MD5-чекcумму содержимого и сравнивает.

Результат для каждой БД: `ok` или `fail` + список причин.

---

## 4. Итоговое состояние после полной миграции

- **Исходные БД** — не тронуты: все изменения схемы в них временные и откатываются.
- **Целевая БД `tenants`** — содержит по одной схеме на каждую БД-арендатора.
- **`migration-state.json`** — хранит полный прогресс миграции:
  - какие БД завершены;
  - какие упали и с какой ошибкой;
  - тайминги и суммарные метаданные.
  Можно безопасно прерывать процесс и продолжать его с того же места.
